# ADR-0007: 文搜文 —— 文档切块进同一 CLIP 空间

- **状态**：Accepted
- **日期**：2026-07-10

## 背景与问题

要支持文搜文（用一段文字检索相关文档）。文档可能很长，而 CLIP 文本编码器有 token 上限
（Chinese-CLIP ~52，SigLIP/OpenAI ~64/77），整篇直接编码会截断丢信息。

## 候选方案

- **方案 A**：整篇文档编码为一条向量。
- **方案 B**：按字符窗口 + 重叠切块，每块一条向量，检索定位到段落。
- **方案 C**：另引专门的长文本 Embedding 模型（如 bge-m3）建独立索引。

## 决策

采用方案 B：字符窗口切块（默认 200 字，重叠 20），复用 CLIP 文本编码器进同一空间。

## 理由

1. 与图/视频/音频**同一个语义空间**，一套索引、一个查询入口即可跨模态。
2. 切块后可定位到具体段落（metadata 带 char 偏移）。
3. 无需引入第二个模型，工程简单。

## 后果

- **正面**：文搜文与文搜图/视频/音频共享索引与代码；段落级定位。
- **负面**：CLIP 文本编码器对**长文语义**的表达弱于专用文本模型；切块会切断跨段上下文。
- **后续影响**：若文档检索成为主场景，可按方案 C 增设专用文本索引（多向量字段/多 collection）。
